AI KNOWLEDGE STUDIO - COMPLETE TECHNICAL DOCUMENTATION
======================================================

1. OVERVIEW
-----------
AI Knowledge Studio is a Flask application for:
- audio/video transcription
- speaker diarization
- live chunk transcription
- transcript summarization
- document ingestion and document Q&A
- transcript Q&A
- transcript semantic search
- translation
- text-to-speech
- query-first agent routing
- plain chatbot interaction with short-term history
- deterministic runtime date/time and math responses

The current system is best described as:
- query-first
- tool-routed
- partially agentic

It is not yet a fully autonomous multi-step planning agent for every request, but it already exposes a unified agent query path and tool registry.


2. STARTUP FLOW
---------------
Startup sequence:
1. Flask app is created.
2. Workspace folders are ensured.
3. MongoDB init runs.
4. ASR model loads.
5. Diarization pipeline loads.
6. Routes are registered.
7. Web UI is served.


3. PRIMARY RUNTIME FLOWS
------------------------
3.1 Plain chat flow
- User types a query without uploaded file / transcript / document context.
- Frontend sends query to `POST /api/agent/query`.
- Router selects `chat_response`.
- `chat_response` passes recent `chat_history` into the LLM layer.
- Deterministic date/time and math handling run before the LLM fallback.
- Backend returns updated `history`.
- Frontend stores that history in `currentAgentChat`.

3.2 Upload + query flow
- User stages a file in the UI.
- No processing starts yet.
- User sends a query.
- Frontend sends file + query to `POST /api/agent/chat`.
- Backend detects media vs document upload.
- File is processed according to the query-driven agent path.

3.3 Transcript workflow
- media is processed
- transcript is generated
- speaker diarization runs
- transcript session is stored
- transcript can be summarized, searched, or used for transcript Q&A

3.4 Document workflow
- document text is extracted
- summary is generated
- text is chunked and embedded
- chunks are stored
- document Q&A uses retrieval + generation

5. AGENT DESIGN
---------------
Current tool registry includes:
- `chat_response`
- `process_media`
- `summarize_transcript`
- `translate_text`
- `answer_document`
- `answer_history`
- `search_history`
- `text_to_speech`

Routing behavior:
- explicit tool in payload is respected first
- otherwise planner / heuristic routing is used
- plain chat queries resolve to `chat_response`
- transcript/document context can shift the route to the matching RAG tools

Important limitation:
- the current router is still mostly single-tool-first routing
- there is limited chaining only in selected cases such as summary + translate


6. PLAIN CHAT MEMORY
--------------------
The normal chatbot now keeps recent conversation history in the frontend.

Frontend:
- `currentAgentChat` stores the current chat thread
- recent turns are sent in `payload.chat_history`

Backend:
- `chat_response_tool()` appends the current turn and returns updated history
- `chat_with_agent()` includes `Conversation History` in the LLM prompt

Scope:
- memory is session-local in the current browser session
- it is reset when transcript/document workflows become active
- it is not yet persisted as a long-term standalone chat history collection


7. DETERMINISTIC UTILITY LAYER
------------------------------
The normal chatbot has a pre-LLM deterministic layer for:

7.1 Date / time
- current day
- current date
- current time
- combined date-and-time phrasing

This is implemented from runtime code using:
- `datetime`
- `zoneinfo`

7.2 Math
- arithmetic
- percentages
- simplification
- factoring
- basic solving

This is implemented with:
- `sympy`

If a question matches the deterministic utility layer, the LLM is bypassed.


8. API ROUTES
-------------
Main routes:

GET /
- web UI

POST /process
- classic media processing endpoint

POST /transcribe_chunk
- live chunk transcription

POST /summarize_text
- classic summary endpoint

POST /process_document
- classic document processing endpoint

POST /translate
- translation endpoint

POST /api/document/ask
- document RAG Q&A

- transcript-history RAG Q&A

GET /api/documents
- list documents

GET /api/documents/<doc_id>
- open document item

GET /api/agent/tools
- list agent tools

POST /api/agent/query
- query-first endpoint for plain chat or tool routing

POST /api/agent/chat
- unified uploaded-file + query endpoint

GET /history
- list transcript history

GET /history/<session_id>
- fetch one transcript session

POST /history/<session_id>/transcript
- save transcript / summary

PATCH /history/<session_id>
- rename transcript history item

DELETE /history/<session_id>
- delete transcript history item

14. TROUBLESHOOTING
-------------------
Chatbot gives guessed date/time answers
- deterministic handling should answer these before the LLM
- verify latest code is running
- verify server restart after code changes

Plain chat forgets earlier turns
- verify frontend is sending `chat_history`
- verify backend returns `history`
- note that memory is only within the current browser session

Ollama-backed replies fail
- verify Ollama is running
- verify `OLLAMA_URL`
- verify model availability

Document Q&A returns weak answers
- verify document text extraction succeeded
- verify embeddings/chunking completed

Translation not working
- verify local NLLB model is available
- verify language codes and offline model path

TTS not working
- verify `gTTS` is installed
- note that gTTS usually needs internet access

MongoDB issues
- verify MongoDB service
- verify connection string


15. CURRENT KNOWN LIMITATIONS
-----------------------------
- plain chatbot history is not yet persisted server-side as a standalone long-term chat store
- agent routing is still mostly single-tool-first, not full autonomous planning for every request
- gTTS depends on runtime connectivity
- some utility answers are deterministic, but not every general knowledge task is grounded
